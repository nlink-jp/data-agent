# RFP: data-agent

> Generated: 2026-04-20
> Status: Draft

## 1. 課題定義

shell-agentは汎用チャットとデータ分析の両立により、実装の難易度上昇と複雑化を招いた。data-agentはこの教訓を踏まえ、「データ分析」に特化したデスクトップGUIツールとして新造する。

JSON/JSONL/CSV/TSV/SQLiteなどの多様なデータを、ケース（案件）ごとに独立したDuckDB/SQLiteに取り込み、対話的なデータ探索・分析・可視化を提供する。DBアクセス層をケースごとにインスタンス化してAPI提供することで、同時アクセス問題を回避する。

単発SQL集計とスライドウィンドウ分析の両方に対応し、テーブル/グラフ表示とMarkdownレポート出力を備える。マルチジョブによる並列分析をサポートし、LLMはVertex AI/ローカルLLM両対応で疎結合に設計する。

ターゲットユーザーは自分および同僚規模の利用。大規模デプロイは想定しない。

## 2. 機能仕様

### コマンド / API サーフェス

`data-agent` でGUI起動（CLIサブコマンドなし）。

**ケース管理:**
- 新規作成、一覧、開く、閉じる、削除、エクスポート
- ケース画面でデータの追加/削除
- ケースごとに独立したDuckDB/SQLiteファイルを管理
- DBアクセス層をインスタンス化してAPI提供
- ケースを「開く」で起動、「閉じる」で破棄（バックグラウンドジョブ実行中の維持方式は要検討）

**分析操作:**
- チャット欄に自然言語で指示 → LLMがSQL生成・実行
- `/sql` コマンドで直接SQLモード切替
- スライドウィンドウ分析
- コンテナ内Python実行（SQLだけでは対応できない分析向け）

**表示:**
- テーブル表示
- グラフ表示（棒・折れ線・円）
- Markdownレポート生成
- ログウィンドウ（システムログを常時表示、処理状況の透明性を確保）

### 入出力

**データ入力:**
- JSON, JSONL, CSV, TSV, SQLite
- ケース管理画面からデータ追加/削除

**分析出力:**
- テーブル表示（GUI内）
- グラフ表示（jviz参考: 棒グラフ、折れ線グラフ、円グラフ）
- Markdownレポート

**レポート出力先:**
- ケースデータとして保存
- ファイル出力
- クリップボード出力

**LLM通信:**
- Vertex AI: Gemini API（ストリーミング対応）
- ローカルLLM: OpenAI互換 `/v1/chat/completions`（ストリーミング対応）

### コンフィギュレーション

`config.toml` で一括管理。

**設定項目:**
- Vertex AI設定（プロジェクトID、リージョン、モデル名）
- ローカルLLM設定（エンドポイントURL、モデル名、APIキー）
- コンテナランタイム設定（Podman/Docker選択、イメージ名）
- 分析設定（トークン予算、コンテキストウィンドウサイズ）

**保存場所:** `~/Library/Application Support/data-agent/`（macOS）

### 外部依存

| 依存 | 種別 | 必須 |
|------|------|------|
| Vertex AI (Gemini) | クラウドAPI | No（LLM二択の一方） |
| ローカルLLM (OpenAI互換API) | ローカルサービス | No（LLM二択の一方） |
| DuckDB | 組み込みDB | Yes |
| Podman / Docker | コンテナランタイム | No（Python分析実行時） |

## 3. 設計判断

**言語・フレームワーク:**
- Go + Wails v2 + React — shell-agentのテクノロジスタックを踏襲。SwiftUIランチャーは不要（shell-agentでも結局未使用）。

**コードベース:**
- 完全新規設計。shell-agentのコード流用はせず、詳細なアーキテクチャ検討から実施する。

**既存ツールとの関係:**
- `shell-agent`（util-series）: 技術スタック（Go + Wails v2 + React）の参考。汎用チャットとの両立が複雑化を招いた教訓。
- `data-analyzer`（util-series）: スライドウィンドウ分析手法の参考。
- `jviz`（util-series）: テーブル/グラフ可視化の参考。

**LLMインターフェース:**
- Vertex AIとローカルLLMの両対応。疎結合に設計し、バックエンド切替を容易にする。
- トークン管理とコンテキスト管理を初期フェーズから設計に組み込む。

**ストリーミング応答:**
- LLM応答はストリーミング表示。処理中のフリーズ感を解消する。

**ログウィンドウ:**
- システムログを単に保存するだけでなく、常時表示するUIパネルとして実装。動作状況の透明性を確保する。

**スコープ外:**
- 汎用チャット機能（データ分析に必要な言語処理は含む）

## 4. 開発計画

### Phase 1: Core — アーキテクチャ・分析基盤

- アーキテクチャ設計（DB層、LLMインターフェース、ジョブ管理の全体設計）
- LLMインターフェース基盤（トークン管理、コンテキスト管理、ストリーミング応答、Vertex AI/ローカルLLM両対応）
- ケース管理（作成/一覧/開く/閉じる/削除）
- データ取り込み（JSON/JSONL/CSV/TSV/SQLite → ケース別DuckDB）
- DBアクセス層のインスタンス化とAPI提供
- 基本チャットUI + 自然言語→SQL生成・実行
- `/sql` 直接SQLモード
- テーブル表示
- config.toml設定基盤
- 基本テスト

**独立レビュー可能**

### Phase 2: Features — 分析機能拡充

- コンテナ実行（Podman/Docker でPython分析コード実行）
- Markdownレポート生成・出力（ケース保存/ファイル/クリップボード）
- スライドウィンドウ分析
- マルチジョブ/バックグラウンド分析のインスタンス化
- ログウィンドウ

**独立レビュー可能（機能単位で個別レビュー可能）**

### Phase 3: Release — 可視化・ドキュメント・リリース

- グラフ表示（棒・折れ線・円）
- README.md / README.ja.md
- CHANGELOG.md
- AGENTS.md
- リリースビルド・配布

## 5. 必要なAPIスコープ / 権限

- **Vertex AI:** ADC認証（Application Default Credentials）、`roles/aiplatform.user` ロール
- **ローカルLLM:** OpenAI互換API、APIキー対応（config.tomlで管理）
- **Podman/Docker:** ローカル実行のみ、特別な権限不要

## 6. シリーズ配置

Series: **util-series**
Reason: shell-agent、data-analyzer、jvizと同系統のデータ処理・分析ツール。util-seriesへの配置は確立済み。

## 7. 外部プラットフォーム制約

| 制約 | 詳細 |
|------|------|
| DuckDB + Wails | `no_duckdb_arrow` ビルドタグ必須（shell-agentで実証済み） |
| Vertex AI | APIレートリミット（大量逐次リクエスト時の429）。トークン管理設計で考慮する |
| Podman/Docker on macOS | 既知のUnixソケット課題あり。ただしdata-agentではコード投入→結果取得のみのため影響なし |

## Discussion Log

1. **ツール名決定:** `data-agent` — データ分析に特化したエージェントツール
2. **shell-agentからの教訓:** 汎用チャットとデータ分析の両立が複雑化を招いた。data-agentではデータ分析に特化することで目的と機能を明確化する
3. **ケースごとDB分離:** 当初はセントラルDBでケース単位にセグメント化する構想だったが、ケースごとにDB自体を分離し、DBアクセス層をインスタンス化してAPI提供する方式に変更。同時アクセス問題（DuckDBのシングルライター制約）を自然に回避できる
4. **DBインスタンスライフサイクル:** ケースを「開く」で起動、「閉じる」で破棄。ただしバックグラウンドジョブ実行中はインスタンスを維持する必要があり、この方式は要検討
5. **LLMトークン/コンテキスト管理:** LLMとの対話がコア機能であるため、Phase 1から設計に組み込む。data-analyzerやshell-agentでのトークン予算過小推定の教訓を反映
6. **コンテナ実行のフェーズ配置:** グラフ表示より優先度が高い（データ分析に直結するため）。Phase 2に配置し、グラフはPhase 3へ
7. **ログウィンドウ:** 単純なログ保存ではなく、常時表示するUIパネルとして実装。処理中のフリーズ感解消とシステム動作状況の透明性確保
8. **Podman macOSソケット課題:** shell-agent/cclaudeでは1Password SSH agent転送が課題だったが、data-agentではコンテナにコードを投入して結果を受け取るだけなので影響なし
9. **ストリーミング応答:** LLM応答をストリーミング表示し、処理中のUI停止感を解消
10. **シリーズ配置:** util-series（shell-agent既配置済み、同系統ツール群と整合）
